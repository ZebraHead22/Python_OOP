# 1️⃣ Путь через контрольные работы (СИД)

В течение семестра проводится **две контрольные работы (СИД)**.

Каждая контрольная оценивается по системе **зачёт / незачёт**.

| Работа | Баллы |
| ------ | ----- |
| СИД 1  | 1     |
| СИД 2  | 1     |

Максимум за контрольные: **2 балла**.

Если студент получает **зачёт за обе работы**, он автоматически набирает:

**80% итогового результата за курс**

Это соответствует **оценке “удовлетворительно”**.

Таким образом, контрольные проверяют **освоение базовых навыков курса**.

## 📋 Требования к решению задач на контрольных работах

При оценке практических заданий (в том числе на промежуточных аттестациях) мы руководствуемся следующими критериями. Они помогут вам понять, как получить максимальный балл.

### 1. Корректность работы программы
- Программа должна выполнять поставленную задачу без ошибок для всех предусмотренных условием входных данных.
- Результат должен точно соответствовать спецификации (формат вывода, поведение в граничных случаях).

### 2. Проверка входных данных (типы и значения)
- **Тип данных**: функция/метод должны проверять, что аргументы имеют ожидаемый тип (или могут быть к нему приведены без потери смысла). При несоответствии — генерировать исключение `TypeError` с понятным сообщением.
- **Корректность значений**: проверка на допустимые диапазоны (например, часы от 0 до 23, возраст человека не более 200 лет, цена товара не отрицательная). При нарушении — выбрасывать `ValueError` (или собственное исключение).
- Пример:
  ```python
  def set_age(self, age: int):
      if not isinstance(age, int):
          raise TypeError("Возраст должен быть целым числом")
      if not (0 <= age <= 200):
          raise ValueError("Возраст должен быть от 0 до 200 лет")
      self._age = age
  ```

### 3. Обработка исключений
- В коде должны быть обработаны потенциально опасные места (работа с файлами, ввод пользователя, деление на ноль и т.п.) с использованием `try/except/else/finally`.
- Исключения не должны «проглатываться» без уведомления. Допускается либо проброс выше, либо вывод информативного сообщения.

### 4. Документирование кода
- Каждый класс и публичный метод должны содержать **docstring** в формате (рекомендуется [Google style](https://google.github.io/styleguide/pyguide.html#38-comments-and-docstrings) или [reStructuredText](https://www.python.org/dev/peps/pep-0287/)).
- В docstring описывается назначение, аргументы (`Args:`), возвращаемое значение (`Returns:`), возможные исключения (`Raises:`).
- Пример:
  ```python
  def calculate_discount(price: float, percent: float) -> float:
      """Возвращает цену со скидкой.

      Args:
          price: Исходная цена (должна быть >= 0).
          percent: Процент скидки (от 0 до 100).

      Returns:
          Цена после применения скидки.

      Raises:
          ValueError: Если price < 0 или percent не в [0, 100].
      """
  ```

### 5. Читаемость и стиль кода
- Соблюдение **PEP 8**: отступы, пробелы, именование переменных и функций (snake_case), классов (CamelCase), констант (UPPER_CASE).
- Код должен быть самодокументируемым: понятные имена переменных, отсутствие «магических чисел» (использовать константы).
- Комментарии поясняют сложные алгоритмы или неочевидные решения (но не пересказывают код).

### 6. Дополнительные требования (для высоких оценок)
- **Эффективность** – разумное использование алгоритмов и структур данных (например, не искать элемент в списке каждый раз, если можно использовать множество).
- **Использование современных возможностей Python** – аннотации типов (typing), dataclasses, перечисления (Enum) и т.д., если это уместно.
- **Локаничность и избегание дублирования** – соблюдение принципа DRY (Don’t Repeat Yourself).
